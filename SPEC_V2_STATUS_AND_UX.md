## Real bugs and divergences I found while verifying

### 1. `resolveContextForSubject` has an N+1 fetch pattern
`relationships.ts:389–395` calls `getRelationshipContexts()` for the coach (one full table fetch), then for **each** direct RC calls `getRelationshipContexts()` again on the downstream person (each = full table fetch + name map rebuild). For Josh's 4 clients that's ~5 round-trips on every note save. Fine today, brutal at 50 clients.

**Fix:** fetch the entire RC table once, build an in-memory `leadId → personIds` index, and walk it in JS. Cache the index on the request. Or use the `unstable_cache` Next.js helper since the data changes infrequently.

### 2. `dashboardLogNoteAction` bypasses the Notes table when a meeting is selected
`actions.ts:122–129` writes to **Coach Session** via `upsertCoachSession()` instead of creating a Note. That means session notes:
- have no `Relationship Context` link,
- don't appear in `getNotesByRelationshipContext()`,
- aren't governed by the spec's privacy boundary,
- live in a parallel store from async notes.

The spec is explicit: every note has Author Person + Subject Person + Relationship Context, with `Note Type = meeting_note` for session notes. **Decision needed:** either start writing to Notes (and stop using Coach Session for session notes), or document this as an explicit v1 deviation. Right now it's an undocumented split-store.

### 3. Multiple stores for the same conceptual data
A coach can write notes about a client in **five** places:
- `Notes` table (proper, RC-scoped)
- `Coach Session.Session Notes` (per-meeting, not RC-scoped)
- `Coach Session.Action Items` (parallel to Tasks)
- `Coach-Person Context.Quick Notes` and `.Family Details` (per-pair)
- `Users.Internal Notes` (per-person, ignores coach)

The profile page reads from all of them and renders them as separate sections. From the coach's POV, the question "where do I write this thought?" has no obvious answer. This was deferred in the gap analysis as v1-acceptable, but the cost is showing up in the UX (see #4 below).

### 4. Two "Edit Profile" buttons on the profile page
`page.tsx:466–472` renders both `EditInlineProfileDialog` and `EditProfileDialog`. They look similar, sit next to each other, and a coach has to know that one edits Title + Internal Notes only and the other edits everything. This is a leftover from incremental migration — needs to be one button.

### 5. "Coaching Context" section shows up to four overlapping subsections
`page.tsx:631–682`:
- Internal Notes (Users field)
- Quick Notes (Coach-Person Context)
- Family Details (Coach-Person Context)
- Relationship Flags (Coach-Person Context)

Conceptually these are all "things this coach knows about this person." Splitting into four labeled subsections forces a coach to memorize which field each thought goes into. Spec answer: collapse them into Notes with `Note Type = general_context`. v1 fix without that migration: collapse the UI into one block.

### 6. "Log Session" / manual meeting creation is still missing
Bug #3 from your original list: still no "Log Session" button on the client detail page. The calendar sync handles automatic meeting creation, but a coach can't manually log a session that wasn't on Outlook (e.g. impromptu phone call). `UserActionsBar` has Create Follow-up Draft, Log a Note, Add Task. No Add Meeting / Log Session.

### 7. "Today" filter on dashboard is timezone-broken
`page.tsx:241–243`:
```ts
const todayStr = now.toISOString().slice(0, 10)
const todayItems = upcomingItems.filter(
  (item) => new Date(item.startTime).toISOString().slice(0, 10) === todayStr,
)
```

This compares UTC date strings. A 7pm Eastern meeting on Friday is at midnight UTC Saturday. The dashboard will show "no sessions today" on Friday afternoon and pop the meeting onto Saturday. Use `formatEastern(iso, { year, month, day })` or equivalent.

### 8. Greeting time-of-day uses server local time
`page.tsx:22–27` uses `new Date().getHours()`. On Render this is whatever the container's TZ is set to (usually UTC). So at 7pm ET you might see "Good morning, Josh." Use the same Eastern formatter, not raw `Date`.

### 9. Console.log spam on every dashboard render
`page.tsx:362–367`:
```ts
clientActivity.forEach(({ user, lastMeeting }) => {
  const email = user.workEmail ?? user.email ?? '(no email)'
  const count = meetingsByUser.get(user.id)?.length ?? 0
  console.log(`[dashboard] ${getDisplayName(user)} | email: ${email} | meetings matched: ${count}...`)
})
```

Plus `[debug]` logs in `meetings.ts` and `[RC]` logs in `relationships.ts`. These run on every server-render of every authenticated page. Wrap in `if (process.env.NODE_ENV !== 'production')` or strip before shipping.

### 10. `Notes.Client` field is now redundant
`saveNoteAction` writes `clientId: subjectPersonId` (line 127) — same value into both fields. Same in `dashboardSaveNoteAction`. The spec uses Subject Person as canonical "who is this about." Either remove the `CLIENT` write or remove the field from Airtable. Keeping a redundant field invites them to drift.

## UX/UI suggestions, ranked by impact

### High impact

**A. Consolidate the "where do I write this?" question.** The Coaching Context block on the profile page is currently four labeled fields backed by three tables. Pick one of these:

  - *Cheap fix (UI only):* render Internal Notes + Quick Notes as a single combined block. Hide Family Details unless populated. Flags become inline badges next to the name.
  - *Proper fix (data + UI):* migrate Quick Notes/Family Details/Flags into Notes (Note Type = general_context), delete `coachPersonContext.ts`, remove the Coaching Context section as a separate area — let the Coach Notes section render them sorted with the rest. This is what the spec wants. Requires a one-off migration script and ~2 days of cleanup.

**B. Add a "Log Session" button.** Manual meeting creation. Required for impromptu sessions, makeup calls, and any case where Outlook didn't fire. Should:
  1. Live next to "Log a Note" in `UserActionsBar`.
  2. Open a dialog with: Date/Time, Duration, Notes (pre-fill empty), checkbox "This was a coaching session."
  3. Create a Meeting record with `Calendar Provider = Manual`, `Meeting Status = Completed`, link to the RC.
  4. Optionally also create a Note attached to the new Meeting in the same submit.

**C. Unify the "Edit Profile" UI.** Delete `EditInlineProfileDialog` OR convert it into a hover-state pencil icon on the relevant fields (Title, Internal Notes). One "Edit Profile" button that opens a single dialog for everything.

**D. Show the coach's view of "Why am I seeing this person?"** When a coach lands on a profile, the relationship context badge says "Coaching · Active since Mar 2024" or shows the warning. Good. But for downstream people (Stephanie reached via Iyan), there's no badge that says "via your coaching with Iyan." Add it. The trail breadcrumb gets you there in URL state, but the badge doesn't reflect it. Useful for coaches working through a multi-level org.

**E. Tasks on profile page should match dashboard filtering.** Dashboard shows Not Started + In Progress. Profile page shows all statuses (Complete tasks too). Either:
  - Group: "Open" (default expanded) + "Completed" (collapsed), or
  - Filter chip: All / Open / Done — defaulting to Open.

### Medium impact

**F. Onboarding org fencing is incomplete.** `NewPersonForm.tsx:189–192`: when `companyId === NO_COMPANY`, the "Reports To" picker shows all users across all companies. Spec Section 7 wants org fencing on this picker. Fix: disable the picker until a company is selected, or filter aggressively when company is unset.

**G. Session notes UX is split across pages.** Coach can edit session notes on the per-session detail page, on the dashboard SessionNotePanel, and read them on the profile page (read-only "Session Notes (from Calendar)"). Make profile page editable inline (or always link out to one canonical edit surface). Pick one source of truth for writes.

**H. Add a contextual empty state on "Coming Up Next" when there's no next meeting.** Currently `nextItem && (...)` — if no upcoming meetings, the section silently doesn't render. Replace with a small CTA: "No sessions scheduled. Check the [calendar][/calendar]."

**I. Make the "Open Client Profile" CTA on Coming Up Next handle unmatched events.** When `clientId` is null (calendar event whose attendee doesn't match any RC), the CTA disappears entirely. Add a CTA: "This person isn't a client yet — Add them."

### Lower impact / polish

**J. Profile page is a 1178-line server component.** Split into composable section components: `<ProfileCard/>`, `<MostRecentSessionSection/>`, `<CoachingContextSection/>`, `<TeamSection/>`, etc. Same render output, drastically easier to maintain.

**K. Avatar color hash is OK but the palette is small.** Add 2–3 more colors. Consider distinct color for clients vs. internal coaches in dashboard avatars so coaches can scan at a glance.

**L. Suspense boundaries on the dashboard.** Dashboard awaits ten parallel fetches before rendering anything. Wrap independent regions (Coming Up Next, Open Tasks, Your Clients) in `<Suspense>` so they can stream. Not a blocker but a perceived-perf win.

**M. Loading skeletons on profile page.** No `loading.tsx` for the profile page (`/users/[id]/loading.tsx` exists — check what it shows). If it's blank, replace with a skeleton card so the layout doesn't jump on load.

**N. Replace `console.log` debug calls with a structured logger.** `pino` or even a wrapper that no-ops in production. Prevents log noise on Render.

**O. The "Last Session" card has a "Has notes" green badge but the next card style differs.** Visual inconsistency. Standardize: same badge component used in both Past Sessions and Last Session.
